## react-shop 작업 가이드

### 1) 목적
- react-shop 작업은 요청 기능 구현 + 구조 정리 + 기본 검증까지 포함해 일관된 품질로 처리한다.
- 별도 요청이 없어도 본 문서의 상시 규칙을 기본 적용한다.

### 2) 기본 원칙
- 코드가 아닌 모든 텍스트는 한글로 작성한다.
- 타입스크립트 기준으로 구현한다.
- 프론트 API 경로의 `/api` 접두사는 임의로 변경하지 않는다.
- 불필요한 대규모 리포맷/리네이밍은 금지한다.
- 모호한 표현은 구현 전에 확인하고, 확정 전 임의 구현을 금지한다.
- 모든 파일은 `UTF-8`(BOM 없음), `LF` 줄바꿈을 유지한다.

### 3) 구현 기준
- 디자인 기준
  - 아이콘: Awesome Font
  - 슬라이더: react-swiper
- 데이터 기준
  - 조회 가능한 데이터는 하드코딩하지 않고 API/DB 조회 결과를 사용한다.
- 코드 작성 기준
  - 모든 메서드, 함수, 클래스에 주석을 작성한다.
  - 메서드/함수 내부 기능별 주석도 작성한다.

### 4) 상시 리팩터링/정리 규칙
- 신규/수정 파일은 `src/domains/{도메인}` + `src/shared/{공통}` 구조를 우선 적용한다.
- 미참조 파일/함수/타입/상수 및 디버그 로그(`console.log`)는 영향 확인 후 함께 정리한다.
- 동일 역할 로직이 2곳 이상이면 공통 유틸/공통 모듈로 통합을 우선 검토한다.
- 중복 서버 호출 로직은 공통 호출 유틸 사용을 우선한다.
- `<img>` 사용으로 ESLint 경고(`@next/next/no-img-element`)가 발생하면 `next/image`로 전환한다.
- React 훅 규칙 경고/오류(`react-hooks/*`)는 수정 범위 내에서 즉시 해소한다.

### 5) 기본 검증 절차 (요청 없어도 수행)
- 코드 변경 후 아래 명령을 기본 실행한다.
  - `npm run lint`
  - `npm run typecheck`
  - `npm run build`
- 완료 기준은 lint/typecheck/build 성공이다.
- 경고가 남으면 원인/영향/후속조치를 결과에 포함한다.
- 환경 제약으로 미실행 항목이 있으면 사유를 명확히 남긴다.

### 6) 화면 테스트 원칙
- 기본 브라우저 직접 오픈 방식 우선 사용, 사용자에게 테스트 장면이 보이는 상태에서 검증 진행
- Playwright는 보조 수단으로만 사용(브라우저 직접 오픈이 불가하거나 재현 자동화/증적이 필요할 때만 사용)
- 브라우저를 단순 수동 확인만 하고 완료 처리하지 말고, 진입/조회/등록/수정/검증 시나리오를 실제 수행한다.

### 7) 브라우저 직접 확인 절차
- 1) 백엔드/프론트 서버를 먼저 기동하고 포트 리슨 상태를 확인한다.
- 2) `http://127.0.0.1:{port}` 또는 `http://localhost:{port}`를 기본 브라우저로 직접 연다.
- 3) 사용자 요청 기능을 화면에서 실제 조작하며 검증한다.
- 4) 결과 보고에는 브라우저 오픈 URL, 실행 명령어, 확인한 동작을 구체적으로 남긴다.
